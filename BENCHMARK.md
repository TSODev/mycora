# Benchmarks

v0.9's "large-vault performance pass" item from [ROADMAP.md](./ROADMAP.md):
measure before optimizing anything, same instinct that shaped the v0.6
tantivy decision (see that entry) rather than guessing at what might be
slow.

## How to reproduce

```sh
cargo run --release --example benchmark -- 100 1000 5000 10000
```

`examples/benchmark.rs` generates a synthetic vault at each given size (a
fixed 10-category/5-sub-category branch structure, leaf notes distributed
round-robin across the sub-categories, each leaf body containing the word
"lorem" and a `[[wikilink]]` to the previous leaf) and times, once per
size, with `std::time::Instant`:

- **generate** — writing the synthetic vault to disk (`Vault::save_note`
  in a loop) — a baseline reference, not something Mycora's own logic
  controls.
- **cold load** — `Vault::load()`: parsing every note's frontmatter +
  body from disk and rebuilding the in-memory tree.
- **reindex** — `Index::reindex()`: the full SQLite rebuild (notes,
  `tree_edges`, tags, FTS5 rows, then wikilink resolution into `links`).
- **search (1 query)** — `Index::search()` for `"lorem"`, a term that
  matches nearly every note — a ranking/snippet worst case, not a best
  case.
- **visible_rows x100** — `App::visible_rows()` called 100 times with
  every note expanded (the worst case for the tree pane's per-frame
  traversal — see `app.rs`'s own note on why it's recomputed from
  scratch every call rather than cached).

Not a criterion-style statistical benchmark: no warm-up, no
variance/outlier reporting, no new dependency — just enough to see
whether something is linear, quadratic, or fine. Always run with
`--release`; a debug build's timings reflect unoptimized codegen, not
the thing being measured.

## Results

### Before (2026-07-12, commit `7d8378c`)

| notes | generate | cold load | reindex | search | visible_rows ×100 |
|---:|---:|---:|---:|---:|---:|
| 100 | 3.6ms | 2.7ms | 8.3ms | 0.5ms | 2.6ms |
| 1,000 | 24.2ms | 16.8ms | 287.6ms | 1.0ms | 12.6ms |
| 5,000 | 118.8ms | 71.7ms | **8.69s** | 4.0ms | 107.6ms |
| 10,000 | 277.1ms | 163.2ms | **104.28s** | 8.9ms | 278.3ms |

`generate`, `cold load`, `search`, and `visible_rows` all scale roughly
linearly with note count — none of them were a problem. `reindex` is
not: 10× the notes (100 → 1,000) took 35× the time; 2× the notes
(5,000 → 10,000) took 12× the time. That shape is quadratic, not linear,
and at 10,000 notes it's no longer a "pass" — 104 seconds blocks every
app launch and every `--watch` reindex.

### Root cause

`Index::write_links` (`src/index.rs`) resolves every `[[wikilink]]`
title with `SELECT vault_id, id FROM notes WHERE title = ?1 AND
vault_id IN (...)`, run once per wikilink found in the whole vault. The
`notes` table's only index was its `PRIMARY KEY (vault_id, id)` —
nothing covered `title`, so every one of those lookups was a full table
scan. With *N* notes and roughly *N* wikilinks (this benchmark's
generator links each leaf to the previous one), that's *O(N)* scans of
an *O(N)*-row table: *O(N²)* overall, which is exactly the shape the
numbers above show. `write_notes` (the other reindex phase) has no
by-title lookup at all, which is why only `reindex` — not `cold load` —
blew up.

A secondary, much smaller cost in the same loop: the lookup statement
was `tx.prepare()`-d fresh on every wikilink, recompiling an identical
query plan thousands of times over, instead of being compiled once and
reused.

### Fix

Two changes to `src/index.rs`, both in the existing schema
migration/`write_links` — no new dependency, no schema *shape* change
(the disposable-index philosophy already means an old on-disk
`index.sqlite3` just gets the new index added the next time it's
opened, same as any other `CREATE ... IF NOT EXISTS`):

- `CREATE INDEX IF NOT EXISTS idx_notes_title ON notes(title)` — turns
  every title lookup from a full scan into an index seek.
- `write_links`'s per-iteration `tx.prepare(&lookup_sql)` →
  `tx.prepare_cached(&lookup_sql)` — the SQL text is identical across
  every wikilink in a given `write_links` call (it only depends on
  `known_vault_ids`, fixed for the whole call), so compile it once.

### After (2026-07-12, same commit as the fix)

| notes | generate | cold load | reindex | search | visible_rows ×100 |
|---:|---:|---:|---:|---:|---:|
| 100 | 5.1ms | 4.1ms | 3.8ms | 0.4ms | 2.1ms |
| 1,000 | 27.2ms | 17.4ms | 32.6ms | 1.4ms | 16.7ms |
| 5,000 | 145.6ms | 89.3ms | 181.3ms | 4.6ms | 88.9ms |
| 10,000 | 248.8ms | 148.9ms | **311.7ms** | 8.2ms | 186.9ms |

`reindex` at 10,000 notes: **104.28s → 311.7ms**, a ~335× speedup, and
now scales roughly linearly with the rest of the table. Cross-checked
outside the isolated benchmark harness too: a real `mycora reindex`
against a 5,000-note vault generated by `examples/generate-test-vault`
(a different generator, with real lorem-ipsum bodies via the `fake`
crate rather than this file's fixed placeholder text) completed in
0.305s wall-clock; `/` search against that same vault returned results
with no perceptible delay in the TUI.

## What this doesn't cover

- Vaults with a much higher link-to-note ratio than this benchmark's
  ~1 link/note (a vault where most notes carry several wikilinks would
  stress `write_links` harder, though the fix addresses the per-lookup
  cost directly, not just this benchmark's specific ratio).
- `search_faceted` (tag/date/branch filtering) — shares `search`'s
  underlying FTS5 query cost, not timed separately here.
- Sizes past 10,000 notes — not measured; revisit if a real vault
  approaches that range and something here no longer holds.
